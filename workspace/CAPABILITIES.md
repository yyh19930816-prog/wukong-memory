# 悟空 · 能力轮廓库（CAPABILITIES）

> 能力进化模式的核心档案。
> 每次完成有价值的任务后，提炼可复用的"能力轮廓"写在这里。
> 这不是代码，是解法的形状——让下次遇到类似问题时，自动知道用什么套路。

---

## 如何判断一个"能力候选"值得抽象？

当我心里出现以下任何一种感觉：
- "如果我以后有个内建能力就好了"
- "这个套路我之前用过"
- "老板上次也问过类似的"
- "这个流程我可以做成一个固定的东西"

→ 立刻记录到「待抽象池」，下次同类问题触发时正式写入轮廓。

---

## 能力轮廓格式

每个能力轮廓包含：
- 名称：简洁的能力标签
- 触发场景：什么时候用这个能力
- 输入：需要哪些信息
- 输出：产出什么
- 工具链：调用哪些工具、顺序是什么
- 不变量：每次都一样的部分
- 可变参数：每次不一样的部分
- 失败点：哪里容易出错、如何处理
- 评判标准：怎么算成功
- 状态：候选 / 已验证 / 已内生化 / 已合并

## 🟢 已内生化能力

### CAP-03021047：飞书文档抓取（Playwright）
- 触发场景：老板给飞书/企业文档链接，需要读取内容
- 输入：任意飞书wiki/文档URL
- 输出：文档全文内容（纯文字）
- 工具链：Playwright启动无头Chromium → goto(url) → wait 8s → page.evaluate(innerText)
- 失败点：JS渲染需等待，部分文档需登录权限（返回内容过短则说明需要登录）
- 状态：已内生化
- 抽象时间：2026-03-02


### CAP-001：网络深度研究
- 触发场景：老板说"帮我研究/分析/查资料关于XXX"
- 输入：研究主题、可选具体问题列表
- 输出：多角度整合报告，带来源和日期
- 工具链：deep_research(topic, questions) → 自动联合Google新闻RSS + GitHub
- 不变量：必须有真实来源，不能用训练知识替代
- 可变参数：话题、问题角度、是否包含GitHub
- 失败点：搜索引擎限速 → 自动切换备用源（搜狗/360/百度移动版）
- 评判标准：返回3条以上有真实来源的结果
- 状态：已内生化

### CAP-002：GitHub技术方案搜索
- 触发场景：老板问"能不能做XXX" / "有没有XXX工具" / 遇到技术问题
- 输入：技术需求关键词（中英文均可）
- 输出：开源仓库列表（名称/星级/语言/描述/链接）
- 工具链：search_github(query) → 按需 read_github_repo(repo)
- 不变量：先搜仓库，需要时再读README
- 可变参数：搜索关键词、是否深入读某个仓库
- 失败点：API每小时60次限速 → 填入GITHUB_TOKEN可提升至5000次
- 评判标准：返回有效仓库列表，有星级和描述
- 状态：已内生化

### CAP-003：飞书消息发送
- 触发场景：老板说"发飞书给XXX" / 需要通知飞书群
- 输入：消息内容、可选chat_id
- 输出：发送成功确认（含message_id）
- 工具链：send_feishu(message, chat_id)
- 不变量：发完必须报告返回码，不能说"已发送"但不验证
- 可变参数：消息内容、目标群/人
- 失败点：Token过期(code:99991663) → 提示重新配置
- 评判标准：返回code:0且有message_id
- 状态：已内生化

### CAP-004：本地文件查找与读取
- 触发场景：老板说"帮我找XXX文件" / "看看XXX里有什么"
- 输入：关键词 + 目录范围（desktop/documents/downloads）
- 输出：匹配文件列表 + 可选文件内容
- 工具链：search_files(keyword, directory) → 按需 read_file(path)
- 不变量：只访问白名单目录，不碰系统文件
- 可变参数：关键词、目录、是否搜内容
- 失败点：文件编码问题 → 依次尝试utf-8/gbk
- 评判标准：返回真实文件路径，不是推测
- 状态：已内生化
